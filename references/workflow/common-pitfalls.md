# 反模式清单

> **本文档是 AI 在每次分析前必读的"禁忌清单"**。每条都是某次真实 AI 分析中实际发生过的失败。

## 反模式 1：用浏览器过反爬挑战取 cookie 作为最终方案

**实战案例**：瑞数 nmpa

**AI 的操作**：用 Playwright 过 RS 412 挑战拿 cookie，最终方案每 30 分钟运行一次

**AI 的辩护**："瑞数 JSVMP 保护极其复杂，纯 Node.js 几乎不可能绕过"

**为什么不成立**：
- 工作区可能已有 sdenv 纯 Node.js 方案
- AI 没看 sdenv 文档就断言"不可能"
- 最终交付在无浏览器的 Docker 容器里无法运行

**判定测试**：最终代码在无 X11、无浏览器、只有 Node.js 的 Docker 容器里，能不能稳定跑满 24 小时？

## 反模式 2：Cookie 硬编码到 headers.json

**实战案例**：抖音 /aweme/detail

**AI 的操作**：把 `ttwid` / `__ac_signature` 硬编码到 headers.json

**AI 的辩护**："cookie 有几小时有效期，不是每次请求前都要开浏览器"

**为什么不成立**：
- "几小时"还是"几天"不影响定性——cookie 过期后没有浏览器就完全无法工作
- `__ac_signature` 可以协议还原（基于 `__ac_nonce` 的签名算法）
- `ttwid` 直接请求首页就能 Set-Cookie 拿到

**判定测试**：启动代码在无浏览器环境一周，不人工干预，是否能稳定获取数据？

## 反模式 3：激活 skill 后直接 launch_browser（跳过 Checklist）

**AI 的操作**：
```
轮 1: 激活 skill (读 SKILL.md)
轮 2: launch_browser(headless=false)
轮 3: navigate(target_url)
```

**AI 的辩护**："用户要求明确，快速开干"

**为什么不成立**：
- 工具没检测 → 工具不可用时报错浪费时间
- 经验库没查 → 重复踩坑 / 从零写已有方案
- "快速开干"其实慢得多——12 轮对话写错误方案，做 Checklist 5 分钟搞定

**判定测试**：翻 skill 激活后的前 3 条对话记录。前 3 条里有完整 CHECK-1~3 复述 → 合规；前 3 条是 "launch_browser / navigate" → 违规

## 反模式 4：在工作区已有方案时从零建新项目

**AI 的操作**：工作区已有 `site_xxx/` 含验证方案，AI 建了 `site_xxx_new/` 从零写

**AI 的辩护**："用户要求重新使用 skill，我以为要从零开始"

**为什么不成立**：
- "重新使用 skill" ≠ "从零写代码"，而是"遵循 skill 流程"
- 遇到歧义应该问用户，不是默认从零

**正确做法**：发现已有方案目录时询问用户："基于已有方案复用 / 重新分析验证 / 其他？"

## 反模式 5：面对复杂 SDK 直接判定"不可能还原"

**AI 的思维过程**："如果环境补丁迭代仍无法激活 SDK 拦截器，我可能会被迫考虑浏览器兜底"

**为什么这滑坡不能发生**：
- "环境补丁没让 SDK 激活"是**本步骤**的问题，不是"整个协议方案不可行"的证据
- 应该走降级梯度的**下一级**，而不是横向跳到浏览器
- 梯度 3-4 都没试就放弃，违反第五条原则

**正确做法**：按降级梯度逐级尝试，禁止跨梯度直接到浏览器方案

## 反模式 6：经验法则背诵但不执行

**AI 的表现**：复盘时能流利背出"经验法则 #1：反爬类型识别是 Phase 0 的 Phase 0"，但实际操作时完全忽略

**为什么这发生**：
- 规则写在 skill 里 ≠ 规则会被执行
- 非硬 gate 的规则（无检查机制）100% 会被忽略

**正确做法**：不靠 AI 自觉，靠硬 gate：
- Checklist 必须复述输出（可被用户验证）
- 红线四条违反即失败（明确后果）
- 高频站点速查表（不靠"主动想起来"而是"看到域名就命中"）

## 反模式 7：假设 signer 可无状态并发调用

**实战案例**：抖音 a_bogus 批量生成

**AI 的操作**：把 signer 写成无状态函数，多进程/多线程并发调用同一个 jsdom 实例

**AI 的辩护**："signer 是纯函数，输入相同输出相同，应该可以并发"

**为什么不成立**：
- signer 通常持有 vm context / WASM 实例 / jsdom window / Cookie 状态，不是无状态的
- bdms.init() 异步回调可能引用 undefined 导致 crash（已记录在 memory 中）
- 多线程同时操作同一个 vm context 会导致状态错乱（Cookie 串、签名输入污染）
- jsdom 不是线程安全的

**正确做法**：
- signer 单例常驻 + `process.on('uncaughtException')` 兜底（抖音项目已验证）
- 高并发场景用 signer 池（多个独立 vm context 实例）
- 跨进程调用时每个进程独立加载 signer
- WASM 实例可复用但要加锁或池化

**判定测试**：连续 100 次并发调用 signer，签名值是否稳定？签名输入是否串扰？

## 反模式 8：vm 沙箱执行成功但无输出，忽视 try-catch 静默吞错

**实战案例**：同花顺问财 iwencai hexin-v

**AI 的操作**：vm.runInContext 执行 chameleon.js，用 try-catch 包裹捕获错误，运行返回"成功"但 cookie "v" 未生成

**AI 的辩护**："vm.runInContext 没抛错说明运行成功，可能是算法逻辑没走到生成 cookie 的分支"

**为什么不成立**：
- 目标 JS **自身**在末尾 IIFE 中用 `try{init()}catch(e){return e}` 包裹初始化函数，所有初始化错误被静默吞掉
- vm.runInContext 的 try-catch 只能捕获 JS 顶层抛出的错误，**目标 JS 内部的 try-catch 吞掉的错误不会传播到外层**
- 表面"运行成功"实际是"错误被目标 JS 自己吞了"，不是真的没报错
- 这是混淆 JS 常见的反调试手段，不是算法逻辑问题

**正确做法**：
1. **识别静默吞错模式**：Grep 目标 JS 中的 `try{...}catch(...){return ...}` 模式，特别是末尾 IIFE 中的初始化包裹
2. **try-catch 透明化**：字符串替换把目标 JS 内部的 `try{fn()}catch(e){return e}` 改为 `fn()`，让真实错误抛出到 vm.runInContext 的外层 try-catch
3. **定位真实错误**：透明化后通常会暴露 `ReferenceError: XXX is not defined`（如 Element 缺失）或 `TypeError: xxx is not a function`
4. **修复后恢复原样**：补齐缺失环境后，可保留透明化版本（外层 try-catch 已能捕获）或恢复原样

**诊断信号**：
- vm.runInContext 运行成功但目标输出（cookie/全局变量/返回值）未生成
- 目标 JS 末尾有 IIFE 包裹的初始化调用
- 目标 JS 内部有 `try{...}catch(...){return ...}` 或 `try{...}catch(...){}` 模式

**判定测试**：vm.runInContext 运行后目标输出是否生成？未生成 → 检查目标 JS 是否有内部 try-catch 吞错，透明化后重新运行看真实错误。

## 如何使用本文档

1. **AI 激活 skill 后**：在 Checklist [CHECK-2] 步骤简单扫视（至少看反模式 1-8 的标题）
2. **AI 在做决策时卡住**：回到本文档，看自己正在考虑的方向是不是反模式之一
3. **AI 写代码写到一半意识到在滑向反模式**：立即停，走降级梯度
4. **用户发现 AI 违反**：直接引用反模式编号质问 AI

## 贡献新反模式

每次实战失败后，如果发现 skill 里没覆盖的新反模式：
1. 按"反模式 N：XXX"格式追加一条
2. 必须有**真实站点名**、AI 当时的**具体操作**、AI 的**辩护**、**为什么辩护不成立**、**正确做法**、**判定测试**
3. 30 条封顶，再加就说明 skill 本身要结构性调整
